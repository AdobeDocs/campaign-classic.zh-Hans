# 迁移到公共云常见问题解答{#dc-faq}



Adobe对旧版数据中心不再授权：Campaign Classic实例必须转移到Public Cloud Amazon Web Services (AWS)。 [了解有关此计划的更多信息](dc-migration.md).

下面是一组有关此项目的常见问题、对您的Campaign环境的影响以及其他有用资源。

对于任何其他问题，您可以联系 [Adobe客户关怀](https://experienceleague.adobe.com/?support-solution=Campaign#support).

## 对基础架构的影响

![](assets/do-not-translate/database.png)

下面列出了对数据库和基础架构的全球影响。

* **数据库是否会改变？ 新数据库的版本是什么？ 将使用什么操作系统？**

   Adobe保留选择并部署最合适的数据库管理引擎的权利，以便在最佳条件下为Adobe Campaign服务提供服务。

   此外，为了保持最佳安全级别，Adobe将不提供与基础架构相关的任何详细信息。

* **是否存在数据丢失的风险？**

   数据库将从旧版数据中心转储并恢复到Public Cloud (AWS)中。 在新数据中心上重新启动后，应用程序将从迁移前的确切状态恢复。 用户不会看到任何差异，只是一些计划任务将被延迟。

* **旧版数据中心与公共云之间的包大小是否有任何差异？**

   我们正在公共云(AWS)中根据当前数据库大小、磁盘大小等使用新包定义进行配置。 例如，如果客户在旧版数据中心拥有一台应用程序服务器，则根据包定义，他们可以在公共云(AWS)拥有两台应用程序服务器。

* **内部版本号或Campaign版本是否会更改？**

   第一步，我们将使用迁移来保持相同的Campaign Classic构建。

   在接下来的步骤中，我们将继续升级到最新的Campaign ClassicGA内部版本。 有关详细信息，请参见[此页面](../../rn/using/rn-overview.md)。

* **解决任何迁移后问题的计划是什么？**

   将在迁移生产系统之前进行大量测试。 但如果有任何问题， [Adobe客户关怀](https://experienceleague.adobe.com/?support-solution=Campaign#support) 仍将是主要联系点。 Adobe已建立了一个专家团队，在需要时提供高级支持。

## 可投放性影响

![](assets/do-not-translate/features.png)

下面列出了对IP、阻止列表、子域和URL的全局影响。

* **如何处理允许列表上的IP？ 允许列表客户是否需要将新的IP地址添加到Campaign传入流量的？**

   Adobe服务器的IP地址将更改。 允许列表因此，客户可能需要在其系统的中添加这些新的IP地址。

   [了解详情](#config) 关于IP on the允许列表。

* **我们将如何处理添加到SFTP/FTP访问允许列表的端口？**

   允许列表 SFTP配置（公钥+上的IP）也将从旧版数据中心移至公共云(AWS)。 客户不需要执行任何操作。

* **我们在更改IP吗？**

   Adobe服务器的IP地址将更改。 允许列表因此，客户可能需要将这些新的IP地址添加到其系统中的。

   [了解详情](#config) 关于IP on the允许列表。

* **如何处理子域委派？**

   现有子域将从旧数据中心移至公共云(AWS)。 此部分将在迁移过程中由Adobe可交付性团队处理。

   Adobe将指导客户进行所需的测试，以确保在迁移后的新公共云(AWS)服务器上配置已启动并正在运行。

* **迁移是否会为跟踪、资源和Web应用程序生成新的URL？**

   否，我们将保留现有URL。

* **子域是否会从Neolane.net更改为campaign.adobe.com？**

   两者 `neolane.net` 和 `campaign.adobe.com` 将在迁移后就位。 简化操作：我们将将neolane.net重定向到Public Cloud (AWS)中的新实例，因此无需客户进行任何更改。

* **IP Warming的计划是什么？**

   首先，Adobe可交付性将评估平台的可交付性状态，并就切换到新IP的计划提出建议

   迁移后无需热备。 这也许是个例外，在这种情况下， [Adobe客户关怀](https://experienceleague.adobe.com/?support-solution=Campaign#support) 会联系客户。

   然而，该计划旨在让这一业务运营变得透明，而不像上线期间那样进行初始升级。

   迁移完成后，Campaign实例将具有完全不同的发送IP。 为确保顺利过渡，Adobe将通过逐步将流量从旧的IP切换到新的IP，对新的发送IP进行升级。

* **允许列表我们是否在上移动URL？**

   是，该文件存储在服务器配置文件中，该文件将从源复制到新实例。

* **我们用于标记通信的子域受委派的子域有何影响？**

   用于营销通信的子域保持不变。 但是，根据实施的不同，需要在客户端执行操作：
   * 如果将子域委派给Adobe（默认），则Adobe会处理所有更改并确保无缝过渡。
   * 如果设置了CNAME（例外），则请求客户端与Adobe协调实施更改。

## 配置和连接影响

![](assets/do-not-translate/maintenance.png)

### 关于IP在允许列表上的注释{#config}

迁移到公共云将为Adobe Campaign应用程序服务器提供新的IP，这样，更改的IP可能会影响Adobe服务器和您的信息系统之间的连接。

![](assets/migration.png)

我们来看看以下两种情况：

* 入站流量：从您的系统或任何其他第三方启动到Adobe Campaign服务器的所有网络活动。 配置将由Adobe处理，然后在迁移期间从旧版复制到公共云。 然后，入站流量的连接将按迁移后的方式保留，并且不需要从客户端执行任何操作

* 出站流量： Adobe Campaign服务器向您的信息系统或任何其他第三方（例如：短信提供商）发起的所有网络活动。 允许列表根据您公司现有的安全策略，更改IP可能需要从您的信息系统或任何其他第三方进行操作

### 全球影响

下面列出了对配置、与其他系统和产品的连接、API和时区的全球影响。

* **迁移是否会影响与外部帐户的连接？**

   是. 例如，第三方集成（短信提供商）应将新的Adobe Campaign允许列表应用程序服务器IP地址添加到。

* **迁移是否会影响使用Genesis连接器的Adobe Analytics连接？ 允许列表如何将Campaign IP地址添加到Adobe Analytics端的？**

   Adobe Campaign应用程序服务器IP地址将更改。 此步骤将在迁移后由Adobe客户关怀团队处理。

* **迁移是否会影响与其他Adobe解决方案(AEM、Target等)的连接？**

   允许列表集成是在和Web服务帐户配置中声明的IP地址的组合。 这将由Adobe客户关怀部门负责和拥有。

   允许列表由于应用服务器IP将发生变化，因此外部解决方案中需要IP地址。 将会提供此信息。 集成的其他部分基于IMS，应按原样工作。

* **对于未连接到组织ID以进行IMS集成的客户，情况如何？**

   将为没有IMS的客户提供一个组织ID：组织ID将附加到其实例。

* **多品牌配置是否受迁移影响？**

   只要将子域和所有相关配置从旧版数据中心正确移动/重定向到公共云(AWS)，我们便不会受到任何影响。

* **API连接是否受迁移影响？**

   Adobe服务器的IP地址将更改。 允许列表因此，客户可能需要将这些新的IP地址添加到其系统中的。

   [了解详情](#config) 关于IP的允许列表。

* **我们是否将确保在迁移后正确设置所有JavaScript内存配置参数？**

   我们会将实例配置从旧版数据中心复制到公共云(AWS)，因此在迁移后将保留这些值。

* **访问某些文件扩展名是否存在任何风险？**

   客户可能希望允许将字体文件、outlook会议文件加载到公共资源文件夹中。 此配置是在当前状态下完成的 `config-<instance>.xml` 文件。 这将与配置文件一起复制。

* **新服务器上的时区是否正在更改？ 客户能否保留其当前时区？**

   它可能会根据新服务器的位置而更改。 但是，客户将能够保留其当前时区。

   [了解详情](../../workflow/using/managing-time-zones.md) 关于Adobe Campaign Classic v7中的时区管理。


## 安全性和权限

![](assets/do-not-translate/security.png)

通过此次迁移到公共云(AWS)，客户环境将保持最新，并满足所有必要的安全要求。 这包括：

* 定期提供最新的操作系统和安全修补程序
* 隔离每个客户的基础架构
* 针对支持云基础架构（如负载均衡器、网络安全规则和存储加密）的受管安全和审核审查。

下面列出了对权限、证书和SFTP访问的影响。

* **我们是否要将所有证书移动到新服务器？**

   是，所有证书都将在此迁移过程中移动。

* **我们是否需要向客户请求新的STP访问密钥？**

   否，Adobe将像在新服务器上一样复制SFTP访问密钥。

* **如何处理SFTP权限？**

   我们正在确保新的SFTP服务器、用户、目录和文件具有完全相同的权限级别。

* **如果无法建立SFTP连接，那么有哪些解决方法/计划可保持客户正常运行？**

   允许列表唯一可能出现的连接问题与客户端的有关。 客户应在非生产环境中添加此测试，以确保它在移至生产环境之前正常工作。

* **允许列表是否有任何数据中心特定的配置需要迁移？**

   允许列表否，没有数据中心特定的配置可管理。

* **我们是否确保在新的环境中成功执行自定义脚本？**

   例如，客户实施可以在工作流中使用自定义脚本（Perl/Shell/Python/Java脚本）来操作文件和文件夹。

   在托管实例上，脚本仅通过JavaScript引擎执行。 这些特定实施可能会导致安全漏洞和升级后问题。 它们不受支持。

* **通过IMS集成，它可以在新实例中正常工作还是需要任何其他配置更新？**

   由于我们保留相同的DNS名称，因此在迁移后，它应该可以正常运行。


## 迁移执行

![](assets/do-not-translate/upgrades.png)

下面列出了迁移过程中的全局影响。

* **我们是否需要计划在迁移期间停止营销活动？**

   Adobe建议在应用程序在旧式数据中心上关闭之前放慢速度并最好暂停所有执行：投放和工作流。 这将简化Cloud Server (AWS)上的重启过程，因为有时间让进程“正常”暂停，并保存任何正在进行的执行状态。

* **我们是否预计Adobe Campaign服务会停机？**

   迁移将伴随不可避免的平台停机。 此计划的目的是指导最大限度地减少此停机时间。

   数据中心之间的数据传输是停机的关键路径。 数据以两种方式存储：

   * 到目前为止，最重要的是数据库
   * 应用程序服务器上的文件（数据导入和导出）

   减小数据库的大小对于加快数据传输至关重要。 建议：

   * 缩短历史数据（投放日志、跟踪日志等）的保留期
   * 删除其他表（投放、收件人、自定义表）上的无用记录


* **迁移实例的估计停机时间是多少？**

   停机时间完全取决于客户数据库和SFTP文件存储的大小。 请联系您的客户关怀联系人以获得估计持续时间。

* **从旧版服务器发送的消息怎么样。 链接是否始终可访问？**

   在迁移运行时，只有一个服务将保持运行状态：电子邮件链接重定向。 所有收件人单击电子邮件后，都将能够访问登陆页面。 但是，不会跟踪这些点击，因此迁移前不久开始的投放的点击率将低于正常值。

* **中源/RT环境呢？**

   MID源和RT的处理方式与任何其他托管的基础架构相同。

* **迁移将按照什么顺序进行？**

   环境将按以下顺序迁移：

   1. 开发环境
   1. 暂存环境
   1. 生产环境
   1. RT环境
   1. 中间源环境

* **什么是回滚计划？**

   回滚计划是将DNS切换回，并将源数据库设置为从只读状态读写。 最终，我们将实现自动化。

* **迁移后，我们是否仍可以访问旧实例？**

   应用程序迁移完成后，没有在旧数据中心上再次运行任何流程的计划。 我们希望可以清除旧数据中心上的所有数据（临时备份除外），直到计划的备份过程在公共云(AWS)上运行。

* **迁移到Public Cloud后，每个实例的测试将允许多长时间？**

   根据客户的复杂性，阶段环境与生产环境迁移之间需要至少1周的备份时间。

* **允许列表谁将处理向添加新IP的问题？**

   Adobe允许列表客户关怀团队将负责通过将新IP添加到来确保客户和任何第三方可以访问新IP系统。
